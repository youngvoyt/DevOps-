## Описание проекта

### Бизнес-задача

Предсказание дефолта заемщиков банка.  
Задача аналитики — оценивать вероятность дефолта клиента на основе его анкетных, социально-демографических и финансовых характеристик, чтобы:

- **снизить кредитные риски банка**;
- **повысить качество скоринговых решений**;
- **выявлять группы повышенного риска** до выдачи кредита.

### Проектная задача (Data Science)

В рамках ЛР №1 реализуются первые шаги проекта:

- **описание структуры данных** в файле `data/data_dictionary.md`;

- оформление структуры репозитория под аналитический проект.

## Файл словаря данных `data/data_dictionary.md`

Словарь данных описывает ключевые поля датасета:

- идентификаторы (`client_id`, `application_id`);
- целевой признак `target_default` (0 — нет дефолта, 1 — дефолт);
- базовые характеристики клиента (`gender`, `age_years`, `marital_status`, `education_level`);
- финансовые показатели (`income_total`, `credit_amount`, `credit_term_months`);
- дополнительные признаки риска (`has_car`, `has_mortgage`, `overdue_30d` и др.).

Файл несколько раз дорабатывался в ветке `dev`, что позволяет:

- проследить **эволюцию структуры признаков**;
- продемонстрировать использование `git diff` для анализа изменений.

**Пример сравнения двух версий словаря данных:**

```bash
# История изменений только для словаря данных
git log --oneline -- data/data_dictionary.md

# Сравнение двух конкретных версий (хеши из команды выше)
git diff 6c08329 16f1f3b -- data/data_dictionary.md
```

## Скрипт загрузки данных `src/loader.py`

Файл `src/loader.py` содержит простую заглушку для загрузки локального CSV-файла:

- проверяет наличие файла в подкаталоге `data/`;
- при отсутствии пробует загрузить файл из корня проекта;
- выводит информационные сообщения о пути к файлу.

Скрипт использовался для моделирования **конфликта при слиянии веток**:

- одна ветка меняла путь к файлу (`data/course_project_test.csv`);
- другая — оставляла загрузку из корня (`course_project_test.csv`);
- при merge возник конфликт, который был разрешён объединением логики.

## Стратегия ветвления (Branching)

В рамках лабораторной реализована следующая схема веток:

| Ветка                 | Назначение                                   |
|-----------------------|----------------------------------------------|
| `main`                | основная стабильная ветка                    |
| `dev`                 | ветка разработки                             |
| `feature/data-loader` | разработка и доработка скрипта загрузки данных |

Основной рабочий процесс:

1. Инициализация репозитория на ветке `main`.
2. Создание ветки `dev` от `main`.
3. Создание ветки `feature/data-loader` от `dev`.
4. Работа над `loader.py` в `feature/data-loader`.
5. Слияние `feature/data-loader` → `dev` с **конфликтом** в `loader.py` и его разрешением.
6. Отправка веток на GitHub и оформление Pull Request `dev` → `main`.

## Демонстрация Git (Лабораторная работа №1)

### Ключевые операции Git

1. **Инициализация репозитория и базовая структура**
   - `git init -b main`
   - добавление `data/`, `src/`, `notebooks/`, `docs/`, `.gitignore`, `README.md`, `data/data_dictionary.md`.

2. **Создание веток**
   - `git checkout -b dev`
   - `git checkout -b feature/data-loader`

3. **Симуляция конфликта при слиянии**
   - изменения одной и той же строки в `src/loader.py` в ветках `dev` и `feature/data-loader`;
   - команда слияния:  
     ```bash
     git checkout dev
     git merge feature/data-loader
     ```
   - Git зафиксировал конфликт в `src/loader.py`.

4. **Разрешение конфликта**
   - ручное редактирование `src/loader.py`: объединение логики поиска файла в `data/` и в корне;
   - фиксация результата:
     ```bash
     git add src/loader.py
     git commit -m "fix: resolve merge conflict in loader"
     ```

5. **Работа со словарём данных и `git diff` (вариант 10)**
   - доработка `data/data_dictionary.md` (добавлены признаки риска: `has_car`, `has_mortgage`, `overdue_30d`);
   - сравнение версий файла:
     ```bash
     git log --oneline -- data/data_dictionary.md
     git diff 6c08329 16f1f3b -- data/data_dictionary.md
     ```

6. **Удалённый репозиторий и Pull Request**
   - добавление origin:
     ```bash
     git remote add origin https://github.com/youngvoyt/DevOps-.git
     git push -u origin main
     git push -u origin dev
     git push -u origin feature/data-loader
     ```
   - создание Pull Request `dev` → `main` через интерфейс GitHub и выполнение merge.
